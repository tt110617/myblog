                          # 0.参考

                          # 1.为什么使用 MQ（好处）

                          优点：解耦、异步、削峰

                          1.解耦：系统A需要发送消息到系统B、C、D，若此时系统C不需要了呢？系统E要加入进来呢？这样的话系统耦合度高。我们可以使用 MQ，系统A发消息到 MQ 中，系统B、C、D、E谁需要消息的话就去 MQ 中消费，这样就实现了系统间的解耦。

                          面试技巧：你需要去考虑一下你负责的系统中是否有类似的场景，就是一个系统或者一个模块，调用了多个系统或者模块，互相之间的调用很复杂，维护起来很麻烦。但是其实这个调用是不需要直接同步调用接口的，如果用 MQ 给它异步化解耦，也是可以的，你就需要去考虑在你的项目里，是不是可以运用这个 MQ 去进行系统的解耦。在简历中体现出来这块东西，用 MQ 作解耦。

                          2.异步：系统A调用系统B中的方法，B的执行时间很长，但是A又需要知道B什么时候执行完，没有 MQ 之前的做法是A每隔一段时间去调用B的查询 API，或者是A提供一个回调接口，这两种方法都不是很优雅，我们引入 MQ 之后，B执行完成后发送消息到 MQ 中，然后由 MQ 去通知A方法已经执行完成。

                          3.削峰：减少高峰时期对服务器的压力。

                          缺点：

                          1.系统可用性降低：MQ 宕机

                          2.系统复杂度提高：如何保证消息不丢失？如何保证消息不被重复消费？如何保证消息的顺序性？

                          3.一致性问题：A系统执行成功就返回了，但是B系统执行失败了怎么办？

                          # 2.Kafka、ActiveMQ、RabbitMQ、RocketMQ 有什么优缺点？

                          Kafka：用于 大数据，有日志方面的需求。

                          ActiveMQ：不常用，社区活跃度低。

                          RabbitMQ：时效性微秒级，比较常用。

                          RockerMQ：经历过双十一等活动的考验，适用于有 并发 场景的需求。

                          # 3.如何保证消息的高可用？

                          RabbitMQ有三种模式：单机模式、普通集群模式、镜像集群模式

                          1.单机模式

                          本地环境下测试着玩的。

                          2.普通集群模式

                          启动多个 RabbitMQ 实例，创建队列时只会在一台机器中，其他队列保存着这个队列的元数据（可以找到队列所在的实例），当你在其他机器访问这个队列时，会从拥有这个队列的机器中拉取队列。

                          没有做到真正的高可用，拥有这个队列的机器宕机时，也就无法通过别的机器访问到队列了。此外，还有数据拉取的开销。

                          主要是 提高吞吐量，让多个节点服务某个队列。

                          3.镜像集群模式

                          高可用，队列的元数据和消息都会存在于多个实例上，每次写队列的时候，会自动把消息同步给其他实例。

                          如何开启？

                          在后面添加镜像集群的策略，可以选择将消息同步给所有实例，也可以选择指定的实例。

                          缺点：

                          1.性能开销大，消息需要同步到所有的实例上。

                          2.没有做到分布式，假设你的队列负载很重，你新增实例，这个实例也包含了这个队列的所有数据，没有扩展性。

                          # 4.如何保证消息不被重复消费？或者说，如何保证消息消费的幂等性？

                          当消费者消费完消息，但还没 ack 时宕机了，MQ 不知道这条消息已经消费，可能会重试，就有可能造成重复消费问题。

                          重复消息不可怕，重要的是要保证 消息消费的幂等性。

                          幂等性：多个相同的请求，需要保证对应数据不会变，不能出错。

                          思路：

                          1.插入之前判断数据是否已经存在，存在就话就更新。

                          2.数据库添加唯一约束，保证多个重复数据只能插入一次。

                          3.如果是写入 Redis，那么 set 命令天然具有幂等性。

                          4.每次发消息时，生成一个全局唯一ID，消费者消费之前，根据 ID 查询 Redis, 如果不存在，说明还没有被消费，此时可以消费消息，之后把 ID 写入 Redis 中；已经存在的话就不处理。

                          # 5.如何保证消息的可靠性传输？或者说，如何处理消息丢失的问题？

                          消息丢失可能出现在 生产者、MQ、消费者端。

                          1.生产者端丢失消息

                          开启 发布确认 模式，RabbitMQ 处理完消息，会调用生产者的 ack 回调接口，RabbitMQ 没能处理消息，会调用生产者的 nack 回调接口。

                          发布确认有三种模式：

                          • 单个确认
                          • 批量确认
                          • 异步确认

                          2.MQopen in new window 丢失消息

                          开启 RabbitMQ 持久化

                          • 创建队列时声明持久化
                          • 发送消息时声明持久化

                          开启持久化也有可能丢失消息，比如 RabbitMQ 消费完消息后，准备写入磁盘，但这时候宕机了，导致少量数据丢失。

                          我们可以搭配发布确认模式，在写入磁盘后才 ack。

                          3.消费者丢失消息

                          使用 消息应答。消费者消费消息后，手动 ack。

                          # 6.如何保证消息的顺序性？

                          先看顺序会错误的场景 binlog 同步：

                          执行增改删，生成 3 条 binlog 语句，发送到队列中，一个队列对应多个消费者，若消费者2执行较快，也就是修改操作先完成了，这样的话顺序就乱了。

                          解决：

                          1.拆分成多个 queue, 一个 queue 对应一个消费者

                          2.一个 queue 对应一个消费者，消费者内部使用 内存队列 做排队，然后分发给不同的 worker 处理。

                          # 7.如何设计一个消息队列？

                          • 首先这个 mq 得 支持可伸缩性 吧，就是需要的时候 快速扩容，就可以增加吞吐量和容量，那怎么搞？设计个分布式的系统呗，参照一下 kafka 的设计理念，broker -> topic -> partition，每个 partition 放一个机器，就存一部分数据。如果现在资源不够了，简单啊，给 topic 增加 partition，然后做数据迁移，增加机器，不就可以存放更多数据，提供更高的吞吐量了？
                          • 其次你得考虑一下 这个 mq 的数据要不要落地磁盘 吧？那肯定要了，落磁盘才能保证别进程挂了数据就丢了。那落磁盘的时候怎么落啊？顺序写，这样就没有磁盘随机读写的寻址开销，磁盘顺序读写的性能是很高的，这就是 kafka 的思路。
                          • 其次你考虑一下 你的 mq 的可用性 啊？这个事儿，具体参考之前可用性那个环节讲解的 kafka 的高可用保障机制。多副本 -> leader & follower -> broker 挂了重新选举 leader 即可对外服务。
                          • 能不能支持数据 0 丢失啊？可以的，参考我们之前说的那个 kafka 数据零丢失方案。

                          mq 肯定是很复杂的，面试官问你这个问题，其实是个开放题，他就是看看你有没有从架构角度整体构思和设计的思维以及能力。确实这个问题可以刷掉一大批人，因为大部分人平时不思考这些东西。
